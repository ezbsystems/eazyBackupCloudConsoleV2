e3 Cloud Backup — UI Component Guide

Live Progress Redesign

Live Backup Monitoring Page — Component Specification

A streamlined live backup monitoring page that replaces the cluttered hero banner, oversized progress section, and toggle-hidden details with a compact progress strip, always-visible run details, and a log panel designed for high-volume output with newest-first ordering and pagination.

Section 01 — Analysis

Current Problems & Solutions

The current page treats every element equally. The redesign establishes a clear visual hierarchy: progress first, stats second, details third, logs dominant.

Remove / Replace
Hero banner. The raised card with job name, status, AGENT badge, and JOB badge. All redundant — the job name becomes the page title, agent and job type move to the details row, status sits next to the progress bar.
"Back to Runs" button. Replaced by a standard breadcrumb at the top of the page.
60px progress percentage. Oversized for the information it conveys. Reduced to 36px — still scannable but doesn't dominate the page.
Details/Logs toggle pills. Removed. Details are always visible. Logs are always visible. No toggling needed.
"Now Backing Up" panel. Replaced with a single inline "Currently processing" line inside the progress strip. Only visible during active runs.
New Approach
Vertical stack, four zones. Breadcrumb + header → Progress strip → Details row → Log panel. Each zone has a single job.
Slim progress bar (6px). Full width, with a subtle gradient and pulse animation when running. The percentage and stage sit above it, stats below it — all in one card.
Stats consolidated. Speed, Processed, Uploaded, Files, Duration, and ETA all live in a 6-column grid under the bar. No separate cards.
Newest-first logs. Most recent entries at the top. Capped at 200 visible lines per page with pagination to browse older entries. Solves the large-log problem.
Log toolbar. Compact bar with search, pause, copy, and clear — all in one row. Not floating buttons.

Section 03 — Log Strategy

Log Volume Strategy

For long-running backups, logs can reach thousands of lines. The design addresses this with newest-first ordering, a 200-line visible window, and server-side pagination.

Newest-First Ordering
New log entries prepend to the top of the output. The user always sees the most recent activity without scrolling. This is a reversal of the current behaviour where new entries append to the bottom and auto-scroll fights with manual scrolling. If the user is on page 1 (the live page), new entries appear at the top automatically. If the user has navigated to an older page, new entries accumulate on page 1 without disrupting their view.
200-Line Window + Pagination
Only 200 lines are rendered in the DOM at any time. This prevents browser performance issues from thousands of DOM nodes. The footer shows "Showing latest 200 of N lines" with Newer/Older page buttons. Pagination is server-side — clicking "Older →" fetches the next 200 lines from the backend. The page count updates as the total grows. When the job is actively running, the line count on page 1 is auto-refreshed.
Pause behaviour. When the user clicks Pause, new entries stop being prepended to the visible output. A small "paused" indicator appears next to the live dot. Internally, the client continues buffering entries and will flush them when the user clicks Resume. This is much simpler than trying to lock scroll position.
Search. Since search is rarely used, it filters the currently visible 200 lines client-side. For full-text search across all log pages, we'd need a server-side search endpoint — that can be a future enhancement. For now, the search input filters the visible page, which covers the common "I just saw an error flash by" use case.

Failed State

When a backup fails, the progress bar turns red, the stage shows "Failed", and the most recent log entries (which typically contain the error) are immediately visible at the top.

Section 05 — Implementation

Implementation Notes

WebSocket / SSE log streaming. New log lines arrive via WebSocket or Server-Sent Events. The client prepends each new line to the top of page 1's DOM. When the visible line count exceeds 200, the oldest line on page 1 is removed and the total line count increments. This keeps the DOM at a constant size regardless of job duration.
Pagination API. The backend serves logs with GET /runs/{id}/logs?page=1&per_page=200&order=desc. Page 1 is always the newest. The response includes total_lines, total_pages, and the lines[] array. Client-side page navigation triggers a fetch and replaces the eb-live-log-output contents.
Progress updates. The progress percentage, stats, stage, and current file all update via the same WebSocket connection. The eb-live-bar-fill width is set dynamically with style="width: {percent}%". The CSS transition: width 0.5s ease provides smooth animation between updates.
Font stack. The log panel uses JetBrains Mono (loaded via Google Fonts) for superior monospace readability. The --eb-font-mono token should be updated from 'Courier New' to 'JetBrains Mono', 'Courier New', monospace for the log components. This is optional — the design works with Courier New as fallback.
CSS integration. Add all new eb-live-* and eb-log-* classes to tailwind.src.css under @layer components { }. All colours use var(--eb-*) tokens. The @keyframes eb-bar-pulse and eb-pulse-dot animations should be added to the global keyframes section.
