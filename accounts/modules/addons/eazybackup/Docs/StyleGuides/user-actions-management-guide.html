User Actions & Management

Delete, Suspend, Bulk Actions — Users Table & Detail Page

Adds row-level action menus with bulk selection to the Users table, an actions dropdown to the User Detail header, and confirmation modals for suspend and delete operations with explicit impact summaries.

Section 02 — User Detail Page

Detail Page Header — Actions Dropdown

The User Detail header gets an "Actions" dropdown button alongside the existing "+ Create Job" and "Back to Users" buttons. The Reset Password form and Delete User card are removed from the Overview tab body — they now live exclusively in this dropdown and its modals.

Header actions area — with dropdown open
Overview tab cleanup. With Reset Password and Delete User moved to the Actions dropdown (which triggers modals), the Overview tab is now cleaner — it contains only the Update User form (username, email, tenant, status) and the Quota controls. Less scrolling, more focused.
Section 03 — Confirmation Modals

Suspend User Modal

Suspending a user stops scheduled backups and denies manual backups, but keeps the agent connected and the user able to log in. The modal explains this clearly.

Suspend confirmation — single user
Suspend User?
CloudlyBackup
Suspending this user will stop all scheduled backups and deny manual backup runs. The agent will remain connected and the user can still log in.
This is typically used when a customer is behind on billing. You can reactivate the user at any time to resume backup operations.

Delete User Modal

Deleting a user cascades to all agents, jobs, and vaults. The modal lists exactly what will be destroyed and requires typing the username to confirm.

Type-to-confirm safeguard. The Delete button stays disabled (dimmed) until the user types the exact username. This prevents accidental deletions, especially during bulk operations. The eb-confirm-input field has a red focus ring to reinforce the danger context.
Impact summary. The eb-impact-list is populated dynamically from the backend — the modal fetches the user's resource counts before rendering. This ensures the admin sees the real cost of deletion, not a generic warning.

Bulk Delete Modal

When multiple users are selected for deletion, the modal shows the aggregated impact across all selected users.

Section 05 — Implementation

Implementation Notes

Bulk selection state. Use Alpine.js x-data="{ selected: [] }" at the table level. Each row checkbox toggles its user ID in the array. The header checkbox toggles all/none. The bulk bar uses x-show="selected.length > 0". Button labels dynamically include the count.
Impact data fetch. When the Delete modal opens, fire an async request to GET /users/{id}/impact (or POST /users/bulk-impact for bulk) that returns { agents: 6, jobs: 1, vaults: 1, storage_gb: 847 }. Render this into the eb-impact-list. Show a loading spinner in the modal body while fetching.
Suspend backend contract. POST /users/{id}/suspend and POST /users/{id}/reactivate. The suspend endpoint should set a flag that the backup agent checks before starting any manual or scheduled backup. The agent connection itself is not terminated.
Delete backend contract. DELETE /users/{id} with cascading deletion. This should be an asynchronous job for users with large amounts of backup data — return 202 Accepted and show a toast: "User deletion in progress. This may take a few minutes." The table row can show a "Deleting…" state with a spinner.
Existing eb-menu-item.is-warning. This is a new variant following the same pattern as .is-danger. Add to tailwind.src.css: background on hover uses var(--eb-warning-soft), text uses var(--eb-warning-text).
